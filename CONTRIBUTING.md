# CONTRIBUTING — envspec

Дев-инструкция. Перед началом прочитать `CLAUDE.md` и `SPEC.md`.

## Окружение
```bash
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -e ".[dev]"
```

## Цикл разработки
1. Выбери задачу из `ROADMAP.md` (идём сверху вниз по этапам).
2. Если задача меняет публичный API/CLI — сначала обнови `SPEC.md`.
3. Напиши/обнови тест под новое поведение (TDD приветствуется).
4. Реализуй минимально достаточно, чтобы тест прошёл.
5. Прогони полный набор проверок (ниже). Всё зелёное = можно коммитить.

## Проверки (Definition of Done)
```bash
ruff check . && ruff format --check .   # стиль
mypy src/envspec                        # типы (strict)
pytest -q                               # тесты
```
Коммит/PR не считается готовым, пока все три не зелёные и поведение не покрыто тестом.

## Конвенции
- Zero-deps ядро. Новая обязательная зависимость — только с обоснованием в описании PR.
- Качество ошибок — приоритет: имя · ожидалось · получено · как починить.
- Секреты (`secret=True`) никогда не печатаются целиком.
- Python 3.9+, без синтаксиса новее. Строки ≤ 100. Docstring у публичных API.
- Коммиты в стиле Conventional Commits: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:`.
- Семвер: ломающее изменение публичного API/CLI = major.

## Структура тестов
`tests/` зеркалит `src/envspec/`. Минимум на этап — критерии приёмки из `SPEC.md §7`.
Без сетевых вызовов; временные `.env`/файлы — через `tmp_path` фикстуру pytest.
